## 1. 決算評価(`EarningsEvaluation`)の判定ロジック

Rust側で独自の判定ロジックは持たない。スクレイピング元サイトが既に表示している評価をそのまま正規化して転記する。将来評価情報を持たないサイトへ切り替わった場合は`Unrated`として扱う(既存enumで対応可能)。

## 2. tickerの正規化ルール

証券コードは取引所によって接尾辞(`.T`など)の有無が異なるため、**常に接尾辞を除去した形に統一**して扱う(例: `7203.T` → `7203`)。将来的に東証以外の取引所へ対応する可能性を考慮し、接尾辞付きの形式には寄せない。

- DB保存・fingerprint生成・表示のすべてにおいて、この正規化後の値を使う
- ユーザがCSV/フロントエンドから`.T`付きでtickerを入力した場合も、同様に正規化してから保存する(表記揺れによる重複登録を防ぐ)
- 正規化処理は、fingerprint生成と同じくRust側の1関数に集約する(`earnings`クレート、`02-types/earnings.md` 4章「残課題」参照。具体的なシグネチャ・置き場所はPhase 4「domain型 / repository trait」着手時に決定する)

## 3. fingerprintの生成

一覧ページで取得できる情報のみを使い、専用構造体からハッシュ化した値を`fingerprint`として新規/既知の判別キーとする。

- fingerprintの生成(結合・正規化・ハッシュ化)は**Rust側の1つの関数に集約**する(表記揺れによる重複判定防止のため)
- 個別ページ側ではfingerprintを再計算しない。一覧取得時点で確定する不変な識別子として扱う
- DBには決算情報の履歴と並行してfingerprint(ハッシュ化後文字列)を保存する(`01-db-schema.md` 5章)
- ハッシュアルゴリズムはSHA-256を採用する。衝突耐性・速度ともに個人利用規模で十分であり、`sha2`クレートで標準的に扱えるため

### fingerprintに含める項目(TODO#4解決)

現在想定しているスクレイピング対象サイト(kabuyoho)における判別用フィールドの中身を以下の通り決定する。

1. 決算速報のタイトル
2. 決算速報の内容の書き出し(一覧ページ表示範囲まで)
3. 決算評価

### 含めないと決定した項目とその理由

- **証券コード**:一覧ページ上に表示されないため取得不可。ただし銘柄名はタイトル・書き出し中に実質的に含まれており、内容も自然文であるため、上記3項目のみで一意性は十分に担保されると判断した
- **公開時刻**:一覧ページ上での表示形式が「当日中はhh:mm」「翌日以降はyyyy/mm/dd hh:mm」のように変化する。同一の決算速報でも取得タイミングによって異なる時刻表記になり、fingerprintに含めると別レコードとして誤って二重登録されるリスクがあるため除外する

> 本内容は今回想定している対象サイト固有の実装(11章「構造体案」の通りサイト専用モジュール内に閉じる)であり、将来別サイトを追加する場合は当該サイトの一覧ページ構造に応じて別途フィールド構成を検討する。

## 4. 決算評価の後日変更時の挙動

サイト側の決算評価が速報後に変わる(例:速報時`Unrated`→後日`Positive`に更新)場合の挙動を明確化する。

- fingerprintに決算評価を含める設計(3章)のため、評価が変わった場合は別レコード(別fingerprint)として**新規決算と同様に検知・再送信される**
- 逆に、fingerprintに含まれない項目(タイトル・書き出し文言等)が変わった場合は検知されないままとする
- これは仕様上許容する(意図的な仕様であり、不具合ではない)

## 5. earningsテーブルの保持方針

決算情報(`earnings`テーブル、`01-db-schema.md` 5章)は削除しない方針とし、無期限に保持する。

- 将来、データ量が許容できないほど増えた場合に備え、年1回程度古いデータを削除するバッチ処理を追加する案はあるが、優先度は非常に低く将来拡張(`05-future-work.md`)止まりとする

## 6. 一覧→個別ページ問題

サイトは「一覧→個別ページ」と遷移する必要があるが、決算件数が多いため個別ページへ毎回遷移するのはコストが高い。そのため、一覧ページの段階で新規/既知を判別し、個別ページへは新規分のみ遷移する。

## 7. Rust/Python間の責務分担

- **Python**:一覧ページの生データ(判別用フィールド群、個別ページへのURL)をJSONで返すのみ
- **Rust**:生データからfingerprintを組み立て・ハッシュ化し、DBの既知fingerprintと突き合わせて新規/既知を判別

これにより正規化ルールの二重管理を避ける。制御ループ(ページ送り判断含む)はすべてRust(`monitor`)側が持ち、Pythonはスクレイピング特有の状態やロジックを一切持たない。

## 8. 差分検出クエリ

> **TODO#5解決により、当初の「時間ベース/件数ベース」2段階方式から変更している。**

### 見直しの経緯
ページ送り打ち切りロジック(9章)は「既知fingerprintが1件でも検出されたらその時点で打ち切る」というページ単位の判定である。この判定に必要なのは「1日の総件数」ではなく「直近1ページ分を確実にカバーできるだけの既知fingerprint」であり、時間軸(当初案の7日分)は件数が少ない閑散期を補うための代理指標に過ぎなかった。件数ベースの取得件数を最初から「1ページの表示件数より十分大きい値」に設定すれば、時間ベースを介さずとも閑散期・繁忙期どちらも1回のクエリでカバーできる(対象サイトの実態調査:1ページあたり約30件表示、繁忙期は1日100件弱の決算速報)。

### 決定内容
- **時間ベース/件数ベースの2段階方式を廃止**し、常に件数ベース1回のクエリに一本化する
- 取得件数は**100件**とする(1ページの表示件数約30件に対して十分な安全マージン)
- この件数はconfigで管理し、運用しながら調整できるようにする

## 9. ページ送り制御

- 該当ページの全件が新規と判定されたら、次ページを取得する
- 既知のfingerprintが1件でも検出されたら、その時点でページ送りを打ち切る

制御ループはRust(`monitor`)側が持ち、Pythonは`list --page N`のようにページ番号を受け取り、該当ページの生データを返すだけとする(スクレイピング特有の状態やロジックをPython側に持たせない)。

## 10. 処理フロー(まとめ)

```text
1. Rust(monitor)がPythonに `list --page 1` を実行させる
2. Rustが返却された生データからfingerprintを組み立てる
3. DBから既知fingerprintを直近100件(config管理)取得しHashSetとする(8章、TODO#5解決)
4. 一覧のfingerprintとHashSetを突き合わせ
   - 全件新規 → 次ページを取得(1に戻る、pageをインクリメント)
   - 既知が1件でもあれば → ページ送りを打ち切り、新規分を確定
5. 新規分のURLのみ、個別ページへ遷移して詳細情報を取得
6. 個別ページの内容と合わせて `MonitoredEarningsReport` として正規化する(02-types/earnings.md)
```

monitor実行全体(健全性チェック用マーカー行の挿入・削除、`notify_queue`への反映)は`01-db-schema.md` 6章「送信内容(キュー)」の「monitorの処理順序」を参照。

## 12. Python連携方式

`scraper`クレート内で、site専用モジュールごとにPythonプロセスを起動し、`stdout`経由でJSONを受け取る。

```rust
// scraper crate内、site専用モジュール(例: kabuyoho.rs)
pub struct KabuyohoScraper;

#[async_trait]
impl ScraperService for KabuyohoScraper {
    async fn fetch_list(&self, page: u32) -> ScraperResult<Vec<RawEarningItem>> {
        let output = tokio::process::Command::new("python3")
            .arg("scripts/kabuyoho/list.py")
            .arg("--page")
            .arg(page.to_string())
            .output()
            .await?;

        let raw: KabuyohoListOutput = serde_json::from_slice(&output.stdout)?;
        Ok(raw.items)
    }
    // fetch_detailも同様
}
```

Python側はサイトごとに`scripts/kabuyoho/`、`scripts/debug/`のようにディレクトリを分け、クレート内のモジュール分割と対応させる。Python側の出力は以下のように`stdout`へJSONを書き出す形で統一する。

```python
print(json.dumps(output, ensure_ascii=False))
```

`debug.rs`(Phase 0の`debug.py`呼び出し用)も`scraper`クレート内の1実装として位置づけ、Phase 13で実サイト実装に差し替える(`00-overview.md` 7.2節)。
