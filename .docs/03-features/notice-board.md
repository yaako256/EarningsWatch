## 1. 設計方針

管理者がマークダウンで自由に書ける「お知らせ・パッチ通知」(時系列)と「固定ページ(現在の対象サイト、使い方説明書など)」を追加する。

- お知らせ(時系列)と固定ページは、性質は異なるが**1つの`pages`テーブルで共通化**する(`type`列で区別)
- 管理者用マークダウンエディタは1種類実装すれば両方に使い回せる
- ページ作成・編集・削除の権限は、既存の管理者権限判定(`Role::Admin`)をそのまま流用する(専用権限は設けない)

## 2. UI設計

既存のダッシュボードと同じ、YaakoDrive方式のSPA的なサイドバー切り替えUIを採用する(URL遷移なし、ブラウザバック非対応。ユーザー体験を優先した意図的な設計)。

- **固定ページ(`type=static`)**:サイドバーに個別項目として並ぶ。表示順は管理者が手動設定(`display_order`)。ページを新規作成すればサイドバーの項目が増える(MVP内で自由追加可能)
- **お知らせ(`type=blog`)**:サイドバーには「お知らせ」という1つの入口のみがあり、そこを開くと時系列(`created_at`降順)の一覧が表示される。個別投稿の公開/非公開は`is_published`で管理者が制御する

## 3. 閲覧権限

お知らせ・固定ページともに**全ユーザー共通**(ログインしていれば誰でも閲覧可能。フィルタ・通知設定と同様の扱い)。

## 4. マークダウンの扱い

- DBには**素のマークダウン本文をそのまま保存**する(元の文章から追記・編集する運用のため)
- HTML変換は**表示時にフロントエンド側で実施**する(marked.js等のライブラリを想定)。バックエンドはマークダウンをそのまま保存・返却するのみで変換ロジックを持たない(`00-overview.md` 4章の責務分離方針と整合)
- 管理者用エディタはMVPでは**簡易エディタ**(テキストエリアのみ、リアルタイムプレビューなし)とする

## 5. データモデル

`pages`テーブル(新設、`01-db-schema.md` 10章)・`Page`/`PageKind`/`PageType`(`02-types/content.md`)を参照。

| 列名 | 型 | 説明 |
|---|---|---|
| `id` | UUID | 主キー |
| `type` | `page_type` enum(`blog`, `static`) | お知らせ/パッチ通知か、固定ページか |
| `title` | text | タイトル |
| `content_markdown` | text | 素のマークダウン本文 |
| `display_order` | integer, nullable | `static`のみ使用。サイドバー表示順(`blog`はNULL) |
| `is_published` | boolean | 表示/非表示 |
| `created_at` / `updated_at` | timestamptz | 作成/更新日時(`blog`の時系列ソートにも使用) |
| `created_by` | UUID(ユーザID) | 作成した管理者 |

## 6. API設計

型定義の詳細は`02-types/api.md` 11章を参照。

| メソッド | パス | 説明 |
|---|---|---|
| GET | `/api/pages?type=static` | 固定ページ一覧(サイドバー用、`display_order`順)。本文を含まない軽量レスポンス |
| GET | `/api/pages?type=blog` | お知らせ一覧(時系列、`created_at`降順、ページング対応)。本文を含まない軽量レスポンス |
| GET | `/api/pages/{id}` | 個別ページの取得(本文含む) |
| POST | `/api/pages` | 新規作成(管理者のみ、`Role::Admin`で判定) |
| PUT | `/api/pages/{id}` | 編集(管理者のみ、`type`・`display_order`は変更不可) |
| DELETE | `/api/pages/{id}` | 削除(管理者のみ) |
| PATCH | `/api/pages/{id}/order` | `static`の表示順変更(管理者のみ) |

## 7. 表示順(display_order)の並べ替えアルゴリズム

`01-db-schema.md` 10章で確定した通り、`display_order`には一意性制約を設けない。並べ替え操作(ドラッグ&ドロップ)時の具体的なアルゴリズムを以下の通り決定する。

### 決定内容
- ページ新規作成時、`display_order`は既存の最大値に**1000刻み**で加算した値を割り当てる(例:1番目=1000、2番目=2000、3番目=3000)。これにより、通常はどの2項目の間にも十分な空き値があり、間へ挿入する際に他の値を変更せずに済む
- ドラッグ&ドロップで並べ替えた際、フロントエンドは移動先の前後の項目の`display_order`の中間値(例:1000と2000の間なら1500)を計算し、`PATCH /api/pages/{id}/order`で移動した1ページ分のみを更新する(1操作=1 API呼び出し)
- 前後の値が隣接していて中間値が取れない場合(例:1000と1001の間)のみ、バックエンド側で対象範囲を1000刻みに再採番してから保存する(頻度は低いと想定)
- 一意性制約がないため、万一同値が発生しても表示順が不定になるだけでエラーにはならない(致命的な不具合にはつながらない設計)

### 残課題(次節以降で検討)
- 再採番処理(中間値が取れない場合)の具体的な実装(`content`クレートまたは`app`クレートのどちらに置くか)は実装フェーズで決定する
