## 1. 基本方針

- 一括インポートボタンから、ファイルを選択して設定する
- Drag & Dropでも設定する
- 列番号指定ではなく、ヘッダ行の値による設定項目の抽出とし、各ユーザが他の情報も含めた設定を可能にする
- 列名の重複を避けるため、アプリ名(`EarningsWatch_`)を列名に含める
- 空行はスキップ
- 壊れた行/足りない行があれば変更を反映せずエラーを出す
- その他列の内容は無視する
- 一新ではなく、バックエンドで変更部分を検出/反映する

## 2. ファイル形式ごとの数式対応方針

| 形式 | 方針 |
| --- | --- |
| CSV | 値のみを扱う。`=`始まりの数式文字列が含まれていた場合は「CSVでは数式はサポートしていません。Excel(.xlsx)をご利用ください。」というエラーにする |
| Excel(.xlsx) | 数式に対応する。数式セルは計算結果(値)を取得してJSONへ変換する |

## 3. 責務分担

| 層 | 責務 |
| --- | --- |
| フロントエンド | CSV/xlsxの違いを吸収し共通のJSON配列に変換する。数式エラー値(`#REF!`等)の検知、ヘッダ不足など、ファイル・パース由来のエラーをここで弾く |
| バックエンド(二重チェック) | `=`始まり文字列が紛れ込んでいないかの最低限の検知(直接APIアクセス等への保険) |
| バックエンド(業務ルール) | 「明らかにおかしい値」の異常値検知(4章)、必須列の空チェック、重複チェック、GroupNameの自動作成/無効化判定など |

ファイル解析・数式処理そのものはフロントエンド側の責務とし、バックエンド(Rust)はCSV/Excelの違いを意識せず、変換済みのJSONを受け取って処理する。

## 4. 異常値検知閾値(TODO#2解決)

「明らかにおかしい値」の異常値検知について、以下の通り決定する。

### 検知方法
- `validate_import_row`(仮称)のようなバリデーション関数として宣言し、将来的に条件を追加しやすい構成にする
- 現時点では「文字数が異常に長いか」の1条件のみ実装する

### 閾値(configで管理し、変更を容易にする)

| フィールド | 上限文字数 |
|---|---|
| `ticker` | 20文字 |
| `company_name` | 50文字 |
| `notes` | 500文字 |

### 検知時の挙動
- エラー行にはしない。行の登録・インポート自体は継続して許可する
- 警告として検知し、インポート結果(レスポンスDTO)に含める(`ImportWarning`、`02-types/api.md` 8章)

## 5. 全体での一括設定(想定列名)

ユーザ全体のフィルタ設定を一新するタイプのインポート(`POST /api/filters/import`)の想定列名:

- `EarningsWatch_Ticker`: 証券コード
- `EarningsWatch_CompanyName`: 会社名
- `EarningsWatch_GroupName`: フィルタグループ
- `EarningsWatch_Notes`: 備考欄(任意)
- `EarningsWatch_Enabled`: 有効/無効(任意、11章参照)

## 6. グループ毎の一括設定(想定列名)

グループ単位の一括インポート(`POST /api/groups/{id}/filters/import`)の想定列名:

- `EarningsWatch_Ticker`: 証券コード
- `EarningsWatch_CompanyName`: 会社名
- `EarningsWatch_Notes`: 備考欄(任意)
- `EarningsWatch_Enabled`: 有効/無効(任意、11章参照)

対象グループはURLの`{id}`で確定しているため、`EarningsWatch_GroupName`列は持たない。

## 7. 「壊れた行」の定義

- 行全体が空(必須列も含め何も入力されていない) → 無視(エラーにしない)
- 必須列(`Ticker`, `CompanyName`)のうち片方だけ入力されている(不完全な行) → エラー行として扱う

## 8. 全体一括設定固有の挙動

ユーザ全体のフィルタ設定を一新するタイプのインポート(`POST /api/filters/import`)では、グループ名(`GroupName`列)に応じて以下を行う。

- 存在しないGroupNameが指定された → 新規グループとして自動作成
- 既存グループのうち、今回のCSVに1件も含まれなかったもの → そのグループを無効化(`paused_at`セット。削除はしない)
- 追加・無効化が発生した内容はレスポンスDTO(`created_groups`/`paused_groups`)でユーザに伝える

## 9. グループ単位の一括インポート固有の挙動

`POST /api/groups/{id}/filters/import`について。

- 対象グループはURLの`{id}`で確定しているため、グループ名の判別・自動作成・無効化は行わない
- インポート対象の有効な行が0件だった場合は`ApiErrorCode::ImportEmpty`エラーとし、インポート処理自体を中断する(既存フィルタは変更しない)

## 10. フィルタ一括インポートのレスポンス例

インポート結果は専用のレスポンスDTO(`ImportFiltersResponse`、`02-types/api.md` 8章)とし、各項目が初期値(0や空配列)でない場合にフロントエンドで注意表示を行う。

```json
{
  "data": {
    "importedCount": 48,
    "skippedEmptyRows": 2,
    "duplicateCount": 3,
    "errorRows": [],
    "createdGroups": [{ "id": "...", "name": "新春決算グループ" }],
    "pausedGroups": [{ "id": "...", "name": "旧グループA" }],
    "warnings": []
  },
  "error": null
}
```

`createdGroups`/`pausedGroups`は全体一括設定でのみ値が入る(グループ単位インポートでは常に空配列)。`GroupRef {id, name}`のペアで返す点は`仮設計書.md`のレスポンス例(グループ名の文字列配列)から変更されている(`02-types/api.md` 8章、フロント側での後続操作の拡張性を考慮した変更)。

## 11. フィルタの一括有効化/無効化/削除(追加仕様)

グループ単位の一時停止/再開(既存)とは別に、グループ内の個々のフィルタを複数選択してまとめて有効化/無効化/削除できる機能を追加する。

- **一括専用APIを新設**する。フロントエンドでのN回ループ処理ではなく、専用エンドポイントで`filter_id`配列をまとめて受け取る方式とする(通信回数・失敗時の一貫性の観点から)
- メソッドは**POSTに統一**する(複数対象を一括処理する性質上、URLで対象を一意特定するDELETE/PATCHの意味論から外れるため、bodyで`filter_id`配列を渡す操作は全てPOSTとする)

| メソッド | パス | 説明 |
|---|---|---|
| POST | `/api/groups/{id}/filters/bulk-enable` | 選択したfilter_id配列を一括有効化 |
| POST | `/api/groups/{id}/filters/bulk-disable` | 選択したfilter_id配列を一括無効化 |
| POST | `/api/groups/{id}/filters/bulk-delete` | 選択したfilter_id配列を一括削除 |

- 既存の単一フィルタ向け`PATCH /enable`・`PATCH /disable`・`DELETE`はそのまま維持し、単発のトグル操作用として共存させる
- 「グループ内の全フィルタを一括変更したい」場合も、対象filter_idを全件分の配列として渡すことで同じAPIで対応可能とする(全体用・選択用でAPIを分けない)
- この一括操作は`00-overview.md`にある「保存ボタン→PUT全体更新」パターンとは別物であり、送信設定(webhook_url等)の保存フローには影響を与えない。フィルタの有効/無効・削除はこれまで通り「即時反映のアクションボタン」として扱う

型定義は`BulkFilterIdsRequest`/`BulkFilterActionResponse`(`02-types/api.md` 8章)。

### 有効/無効列の追加

一括インポートの想定列名(5・6章)に、フィルタの有効/無効を指定する列がなかったため追加する。

- 列名`EarningsWatch_Enabled`を追加する(全体一括設定・グループ毎一括設定の両方)
- **必須列にはしない**。`notes`列と同様、任意入力の扱いとする
- 値が未入力・列自体が存在しない場合は**エラーにはせず、デフォルトで`enabled = true`として扱う**
- 未入力の行がある場合はインポート結果に警告として表示する(`ImportWarning`、10章のレスポンス例参照)

## 12. フィルタデータのExcel出力(追加仕様)

インポートで使うCSV/Excelと同一の列構成で、現在のフィルタ設定データをエクスポートできるようにする。

### 検討経緯
CSV単独運用には、文字コード起因の文字化け(Mac等での表示崩れ)に加え、tickerのような先頭ゼロを含む可能性のある値が数値として誤解釈される「列の型崩れ」リスクがある。Excel形式であればセルごとに型を明示的に持てるためこの問題を回避できる。一方でCSV/Excel両対応は実装コストが倍になるため、まずはExcel出力のみに絞り、CSV出力は後から追加する方針とした(Excel生成用の行データを流用してテキスト化するだけのため、後付けの実装コストは低いと見込まれる)。

### 決定内容
- **MVPではExcel(xlsx)出力のみ実装**する
- **CSV出力は将来拡張(`05-future-work.md`)に追加**する
- **グループ別シート分けは行わない**(1シートにフラットな一覧として出力する)
- 対象範囲は「全体一括エクスポート」と「グループごとのエクスポート」の両方に対応する
- 出力列は、5・6章のインポート用フォーマットと同一の構成とする(`Ticker`/`CompanyName`/`GroupName`/`Notes`/`EarningsWatch_Enabled`等)

同じ設計方針(CSVの型崩れリスク回避)は決算情報エクスポート(`02-types/api.md` 5章)にも同様に適用されている。

### API設計

| メソッド | パス | 説明 |
|---|---|---|
| GET | `/api/filters/export?format=xlsx` | 全グループの全フィルタを一括エクスポート |
| GET | `/api/groups/{id}/filters/export?format=xlsx` | 指定グループのフィルタのみエクスポート |

- `format`クエリパラメータ(`ExportFormat`、`02-types/api.md` 12章)。MVPでは`xlsx`のみ対応
- 将来`format=csv`を追加する際も、パス・パラメータ構造自体は変更せず、実装のみ追加すれば済む構成としておく

## 13. フィルタ一括インポートのドライラン機能(追加仕様)

一括インポート時に、実際の反映前に結果をプレビューできるようにする。

### 決定内容

インポートのフローを以下の通りとする。

1. ユーザがCSV/Excelファイルをドラッグ&ドロップ
2. バックエンドが解析・検証を行い、結果(`imported_count`/`skipped_empty_rows`/`duplicate_count`/`error_rows`/`created_groups`/`paused_groups`等、10章のレスポンスDTOと同じ形式)を**プレビューとして返す**(この時点ではDBへの反映は行わない)
3. フロントエンドはプレビュー結果を表示し、「実際に適用しますか? Yes/No」を確認
4. Yesが選択された場合のみ、確定APIを呼び出して実際にDBへ反映する

- 既存の`POST /api/filters/import`・`POST /api/groups/{id}/filters/import`に`dry_run`のようなクエリパラメータ(またはボディフラグ)を追加し、`true`の場合はDBへの反映を行わずプレビュー結果のみ返す方式とする
- 確定反映時は、プレビュー時と同じファイル内容を再送信して`dry_run=false`で呼び出す想定とする(サーバ側でプレビュー結果を一時保持する必要がなく、ステートレスに保てるため)

> **注記**: `02-types/api.md` 8章の`ImportFiltersRequest`/`ImportGroupFiltersRequest`の型定義には現時点で`dry_run`フィールドが含まれていない(型定義書の時点から未反映のまま持ち越されていた)。実装時にどちらかのDTOへ追加する必要がある。本設計書末尾の「矛盾点・要確認事項」にも記載する。
