## TODO#3解決: mention_targetsの記法変換ロジック（対応: 8.7, 12.3）

Discord向けのmention_targets記法変換を先に決定する。Slackは今回対象外とし、MVP内でDiscord実装後に拡張する際に別途決定する（TODO#1として残置）。

### mention_targetsの保存形式
DBは`TEXT[]`のまま。要素は以下のtypeを持つ文字列とする。

| type | 形式 | 入力方式 |
|---|---|---|
| user | `user:<discord_user_id>` | フロントで自由入力（IDをそのまま貼り付け） |
| role | `role:<discord_role_id>` | フロントで自由入力（IDをそのまま貼り付け） |
| 特殊系（メンション） | `everyone` / `here` | フロントで選択式（enum的なUIで誤入力を防止） |
| 特殊系（タイムスタンプ） | `time:<style>` | フロントで選択式。styleは以下7種から選択 |

コロンの有無・プレフィックスで機械的にtype判別が可能（`user:`/`role:`/`time:`プレフィックスがあればID・パラメータ系、なければ`everyone`/`here`の固定値）。

### timeスタイル一覧（7種全対応）
| style | 表示例 | 用途 |
|---|---|---|
| `t` | 4:20 PM | 短い時刻 |
| `T` | 4:20:30 PM | 長い時刻 |
| `d` | 3/6/2026 | 短い日付 |
| `D` | March 6, 2026 | 長い日付 |
| `f` | March 6, 2026 4:20 PM | 短い日付時刻（Discordのデフォルト） |
| `F` | Friday, March 6, 2026 4:20 PM | 長い日付時刻 |
| `R` | in 2 hours | 相対時間（Discord側でリアルタイム更新） |

### Discord変換ロジック（notify処理側で実装）
| mention_targets要素 | Discordメッセージ本文への変換 | allowed_mentionsへの追加 |
|---|---|---|
| `user:123` | `<@123>` | `users: ["123", ...]` に追加 |
| `role:456` | `<@&456>` | `roles: ["456", ...]` に追加 |
| `everyone` | `@everyone` | `parse: ["everyone"]` に追加 |
| `here` | `@here` | `parse: ["everyone"]` に追加（`@here`も`everyone`パースフラグで許可される） |
| `time:<style>` | `<t:<unix_timestamp>:<style>>`（送信時刻をUnix秒に変換して埋め込み） | 対象外（メンションではないため`allowed_mentions`不要） |

**注意**: Discordはデフォルトでcontent内の全メンションを発火させてしまうため、`allowed_mentions`で明示的に許可リストを組む必要がある（未指定分は`parse: []`でブロックする安全設計とする）。`time:<style>`はメンションではなく表示形式の埋め込みのため、`mention_enabled`のON/OFFやallowed_mentionsの対象外として扱う。

### Slackについて
今回は対象外。Discord実装完了後、MVP内で拡張実装する際にSlackの記法（Block Kit等）を調査し決定する（TODO#1として引き続き残す）。





## TODO#4解決: スクレイピング判別用フィールド構成（対応: 7.6）

現在想定しているスクレイピング対象サイトにおける`EarningItems`の`fingerprint_item_*`の中身を以下の通り決定する。

### fingerprintに含める項目
1. 決算速報のタイトル
2. 決算速報の内容の書き出し（一覧ページ表示範囲まで）
3. 決算評価

### 含めないと決定した項目とその理由
- **証券コード**：一覧ページ上に表示されないため取得不可。ただし銘柄名はタイトル・書き出し中に実質的に含まれており、内容も自然文であるため、上記3項目のみで一意性は十分に担保されると判断した
- **公開時刻**：一覧ページ上での表示形式が「当日中はhh:mm」「翌日以降はyyyy/mm/dd hh:mm」のように変化する。同一の決算速報でも取得タイミングによって異なる時刻表記になり、fingerprintに含めると別レコードとして誤って二重登録されるリスクがあるため除外する

### 補足
本内容は今回想定している対象サイト固有の実装（7.6節の通りサイト専用モジュール内に閉じる）であり、将来別サイトを追加する場合は当該サイトの一覧ページ構造に応じて別途フィールド構成を検討する。





## TODO#5解決: 差分検出クエリの設計見直し（対応: 7.3）

当初「時間ベース→件数ベース」の2段階方式としていたが、対象サイトの実態調査（1ページあたり約30件表示、繁忙期は1日100件弱の決算速報）を踏まえて設計を見直した。

### 見直しの経緯
7.4節のページ送り打ち切りロジックは「既知fingerprintが1件でも検出されたらその時点で打ち切る」というページ単位の判定である。この判定に必要なのは「1日の総件数」ではなく「直近1ページ分を確実にカバーできるだけの既知fingerprint」であり、時間軸（7日分）は件数が少ない閑散期を補うための代理指標に過ぎなかった。件数ベースの取得件数を最初から「1ページの表示件数より十分大きい値」に設定すれば、時間ベースを介さずとも閑散期・繁忙期どちらも1回のクエリでカバーできる。

### 決定内容
- **時間ベース/件数ベースの2段階方式を廃止**し、常に件数ベース1回のクエリに一本化する
- 取得件数は**100件**とする（1ページの表示件数約30件に対して十分な安全マージン）
- この件数はconfigで管理し、運用しながら調整できるようにする

### 7.3節・7.5節への反映内容
- 7.3節の「1. 時間ベースで取得」「2. 件数ベースで取得し直す」という2段階の記述を、「既知fingerprintを直近N件（config管理、初期値100件）取得する」という単一クエリの記述に置き換える
- 7.5節の処理フロー「3. DBから既知fingerprint（時間ベース→不足時は件数ベース）をHashSetとして取得」の部分も、「3. DBから既知fingerprintを直近N件（config管理）取得しHashSetとする」に修正する





## TODO#6解決: 本番Cookie SecureフラグとTailscale運用の整合性（対応: 11.2）
Tailscale閉域内での運用はHTTP接続のため、Secure=trueにするとブラウザがCookieを送信せずログインが機能しない。既存運用中のYaakoDriveで同様の問題に直面した実績を踏まえ、以下の方針とする。

### 決定内容
- 本番・開発ともに、Secureはconfigで環境ごとに切り替え可能な設定のままとする
- 現時点では**本番・開発の両方でSecure=false**とする（Tailscale自体がWireGuardで通信を暗号化しているため、アプリ側の追加保護は現状不要と判断。12.2節の通信経路保護の方針と整合）
- 将来HTTPS化（18章将来拡張）が完了した時点で、**本番のみSecure=trueへ切り替える**（開発はvite等の都合上false運用を継続する想定）

### 11.2節 Cookie設定表の更新
| 属性 | 本番 | 開発 | 理由 |
|------|------|------|------|
| Secure | **false**（HTTPS化後にtrueへ切替） | false | Tailscale閉域内はHTTP運用のため、trueにするとCookieが送信されずログイン不能になる。HTTPS化後は本番のみtrueに切り替える |

（他の属性 HttpOnly/SameSite/Path は変更なし）

## 追加仕様: フィルタの一括有効化/無効化/削除（対応: 14.11 フィルタAPI）

グループ単位の一時停止/再開（既存）とは別に、グループ内の個々のフィルタを複数選択してまとめて有効化/無効化/削除できる機能を追加する。

### 決定内容
- **一括専用APIを新設**する。フロントエンドでのN回ループ処理ではなく、専用エンドポイントでfilter_id配列をまとめて受け取る方式とする（通信回数・失敗時の一貫性の観点から）
- メソッドは**POSTに統一**する（複数対象を一括処理する性質上、URLで対象を一意特定するDELETE/PATCHの意味論から外れるため、bodyでfilter_id配列を渡す操作は全てPOSTとする）

| メソッド | パス | 説明 |
|---|---|---|
| POST | `/api/groups/{id}/filters/bulk-enable` | 選択したfilter_id配列を一括有効化 |
| POST | `/api/groups/{id}/filters/bulk-disable` | 選択したfilter_id配列を一括無効化 |
| POST | `/api/groups/{id}/filters/bulk-delete` | 選択したfilter_id配列を一括削除 |

- 既存の単一フィルタ向け`PATCH /enable`・`PATCH /disable`・`DELETE`（14.11節）はそのまま維持し、単発のトグル操作用として共存させる
- 「グループ内の全フィルタを一括変更したい」場合も、対象filter_idを全件分の配列として渡すことで同じAPIで対応可能とする（全体用・選択用でAPIを分けない）
- この一括操作は14.4節の「保存ボタン→PUT全体更新」パターンとは別物であり、送信設定（webhook_url等）の保存フローには影響を与えない。フィルタの有効/無効・削除はこれまで通り「即時反映のアクションボタン」として扱う

## 追加仕様: フィルタデータのExcel出力（対応: 14.11 フィルタAPI, 9章）

インポートで使うCSV/Excelと同一の列構成で、現在のフィルタ設定データをエクスポートできるようにする。

### 検討経緯
CSV単独運用には、文字コード起因の文字化け（Mac等での表示崩れ）に加え、tickerのような先頭ゼロを含む可能性のある値が数値として誤解釈される「列の型崩れ」リスクがある。Excel形式であればセルごとに型を明示的に持てるためこの問題を回避できる。一方でCSV/Excel両対応は実装コストが倍になるため、まずはExcel出力のみに絞り、CSV出力は後から追加する方針とした（Excel生成用の行データを流用してテキスト化するだけのため、後付けの実装コストは低いと見込まれる）。

### 決定内容
- **MVPではExcel（xlsx）出力のみ実装**する
- **CSV出力は将来拡張（18章）に追加**する
- **グループ別シート分けは行わない**（1シートにフラットな一覧として出力する）
- 対象範囲は「全体一括エクスポート」と「グループごとのエクスポート」の両方に対応する
- 出力列は、9.4節・9.5節のインポート用フォーマットと同一の構成とする（`Ticker`/`CompanyName`/`GroupName`/`Notes`/`EarningsWatch_Enabled`等）

### API設計（既存 GET /api/filters/export を置き換え）
| メソッド | パス | 説明 |
|---|---|---|
| GET | `/api/filters/export?format=xlsx` | 全グループの全フィルタを一括エクスポート |
| GET | `/api/groups/{id}/filters/export?format=xlsx` | 指定グループのフィルタのみエクスポート |

- `format`クエリパラメータを新設。MVPでは`xlsx`のみ対応（デフォルト値も`xlsx`とする）
- 将来`format=csv`を追加する際も、パス・パラメータ構造自体は変更せず、実装のみ追加すれば済む構成としておく






## 追加仕様: お知らせ板・パッチ通知の設計（新設機能）

管理者がマークダウンで自由に書ける「お知らせ・パッチ通知」（時系列）と「固定ページ（現在の対象サイト、使い方説明書など）」を追加する。

### 設計方針
- お知らせ（時系列）と固定ページは、性質は異なるが**1つの`pages`テーブルで共通化**する（`type`列で区別）
- 管理者用マークダウンエディタは1種類実装すれば両方に使い回せる
- ページ作成・編集・削除の権限は、既存の管理者権限判定（is_admin等）をそのまま流用する（専用権限は設けない）

### UI設計
既存のダッシュボードと同じ、YaakoDrive方式のSPA的なサイドバー切り替えUIを採用する（URL遷移なし、ブラウザバック非対応。ユーザー体験を優先した意図的な設計）。

- **固定ページ（`type=static`）**：サイドバーに個別項目として並ぶ。表示順は管理者が手動設定（`display_order`）。ページを新規作成すればサイドバーの項目が増える（MVP内で自由追加可能）
- **お知らせ（`type=blog`）**：サイドバーには「お知らせ」という1つの入口のみがあり、そこを開くと時系列（`created_at`降順）の一覧が表示される。個別投稿の公開/非公開は`is_published`で管理者が制御する

### 閲覧権限
お知らせ・固定ページともに**全ユーザー共通**（ログインしていれば誰でも閲覧可能。フィルタ・通知設定と同様の扱い）。

### マークダウンの扱い
- DBには**素のマークダウン本文をそのまま保存**する（元の文章から追記・編集する運用のため）
- HTML変換は**表示時にフロントエンド側で実施**する（marked.js等のライブラリを想定）。バックエンドはマークダウンをそのまま保存・返却するのみで変換ロジックを持たない（5章の責務分離方針と整合）
- 管理者用エディタはMVPでは**簡易エディタ**（テキストエリアのみ、リアルタイムプレビューなし）とする

### データモデル：`pages`テーブル（新設）
| 列名 | 型 | 説明 |
|---|---|---|
| `id` | UUID/serial | 主キー |
| `type` | enum(`blog`, `static`) | お知らせ/パッチ通知か、固定ページか |
| `title` | text | タイトル |
| `content_markdown` | text | 素のマークダウン本文 |
| `display_order` | integer, nullable | `static`のみ使用。サイドバー表示順（`blog`はnull） |
| `is_published` | boolean | 表示/非表示 |
| `created_at` / `updated_at` | timestamp | 作成/更新日時（`blog`の時系列ソートにも使用） |
| `created_by` | ユーザID | 作成した管理者 |

### API設計（案）
| メソッド | パス | 説明 |
|---|---|---|
| GET | `/api/pages?type=static` | 固定ページ一覧（サイドバー用、`display_order`順） |

## 補足: notify_queueのライフサイクルとリトライ設計（対応: 7.5, 8.2, 10.4, 13.6, 13.7）

monitor/notifyの実行順序、失敗時の挙動、monitor健全性チェックの仕組みを明確化する。

### notify_queueテーブルの拡張
- `is_monitor_marker BOOLEAN NOT NULL DEFAULT FALSE` 列を追加する
- `fingerprint`列は`NOT NULL`から**NULL許容**に変更する（マーカー行では使用しないため）
- 決算データ行は`is_monitor_marker = false`、健全性チェック用マーカー行は`is_monitor_marker = true`として明示的に区別する

### monitorの処理順序
1. monitor開始時、`is_monitor_marker = true`の行を1行挿入する（実行中である印。決算データ行には触れない）
2. スクレイピングを最後まで実行し、新規決算を全部収集する
3. 収集完了後、既存の決算データ行（`is_monitor_marker = false`）を削除し、新規分を`status='ready'`で一括追加する
4. 最後にマーカー行（`is_monitor_marker = true`）を削除する（monitor正常完了の印として消す）

この方式により、決算データ行への削除・追加は「削除→追加」の間に空になる瞬間があっても、健全性チェックはマーカー行の有無だけで判定できるため、決算データ行が一時的に空になるタイミングと健全性チェックのタイミングが競合しない。

### notify_queueの状態遷移（決算データ行）
- `notify`完了後は物理削除せず、`status='sent'`にするのみとする（次回monitorが削除するまで残る）

### monitor健全性チェック（notify実行開始時）
monitor→notify→monitor→notifyという交互実行フローを前提とすると、`notify`実行開始時に`is_monitor_marker = true`の行が存在する場合、monitorが正常に完了していない（実行中またはハング）可能性を示す。

- この場合、送信を行わず数分待ってから再チェックする、というリトライを複数回試みる
- それでも状況が変わらなければmonitorハングと判断し、開発者向けログ通知経路（10.4節`MemoryLayer`）へ警告を送信する
- スクレイピングは負荷軽減のため時間をかけて実行し、指数バックオフも実装する想定であるため、この健全性チェックはその実行時間も踏まえたリトライ間隔とする

### 個別送信失敗時のリトライ（notify実行内で完結）
- 送信が`failed`になった場合、数分待って再送信を試みるリトライを複数回行う
- このリトライは**当該notify実行内で完結**し、次回notify実行には持ち越さない（次回notify実行時のmonitor健全性チェックとは独立したロジック）
- 最終的にリトライしても失敗した場合は、`notify_history`に`failed`として記録される（14.2節の`notify_send_failed`/`notify_rejected`をそのまま利用）





## 型定義方針（本設計書作成にあたっての共通ルール）

DTO・ドメイン型を厳密に定義する段階に先立ち、プロジェクト共通の型ルールを以下の通り定める。

### ID型
YaakoDriveのnewtypeパターンをそのまま踏襲する。`identity`クレート配下に、以下の統一形式で各ID型を定義する。

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub struct XxxId(Uuid);

impl XxxId {
    pub fn new() -> Self { Self(Uuid::new_v4()) }
    pub fn from_uuid(uuid: Uuid) -> Self { Self(uuid) }
    pub fn as_uuid(&self) -> &Uuid { &self.0 }
}

impl Default for XxxId {
    fn default() -> Self { Self::new() }
}

impl std::fmt::Display for XxxId {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        write!(f, "{}", self.0)
    }
}
```

対象: `UserId` / `GroupId` / `FilterId` / `PageId`（今回追加の`pages`テーブル用）等、DB上UUID主キーを持つ全テーブルに対応するID型を用意する。

### EarningsEvaluationのDB⇔Rustマッピング
- DB側`earnings_evaluation`は全大文字（`POSITIVE`/`NEUTRAL`/`NEGATIVE`/`UNRATED`）のまま維持する
- Rust側はパスカルケース（`Positive`/`Neutral`/`Negative`/`Unrated`）のまま維持する
- sqlxの`#[sqlx(type_name = "earnings_evaluation", rename_all = "UPPERCASE")]`でマッピングする

### mention_targetsのRust表現
DB上は`TEXT[]`のまま保持するが、Rust側の業務ロジックでは判別Union型（enum）にパースしてから扱う。

```rust
pub enum MentionTarget {
    User(String),   // user:<id>
    Role(String),   // role:<id>
    Everyone,        // everyone
    Here,             // here
    Time(TimeStyle), // time:<style>（TimeStyleはt/T/d/D/f/F/Rの7種enum）
}
```

### 数値型の方針
過度な型最適化（u8/u16など）はせず、可読性・安全性を優先する。

| 用途 | 型 | 理由 |
|---|---|---|
| カウント・件数・ページ番号などの非負整数ドメイン値 | `u32` | 意味的に適切 |
| DB `BIGINT`と対応する自動採番ID・カウンタ等 | `i64` | DB側の型に合わせる |
| Rustのスライス・配列アクセス用インデックス | `usize` | 言語仕様に従う |
| 浮動小数点 | `f64` | 標準的な精度 |

### タイムゾーンの方針
- サーバ・DBは常にUTCで統一する（DB `TIMESTAMPTZ`、Rust側`DateTime<Utc>`）
- スクレイピング元サイトの表示時刻（`published_at`等）は**JSTとして解釈してからUTCへ変換**し、DBに保存する
- APIレスポンスは**UTCのまま**返す。JST表示への変換は**フロントエンド側の責務**とする

### DTOの命名規則
- Rust構造体のフィールドは`snake_case`のまま定義し、`#[serde(rename_all = "camelCase")]`を付与してJSONシリアライズ時のみ`camelCase`に変換する
- Reactフロントエンド・Rustバックエンドそれぞれの標準的な命名規則を崩さずに両立させる


## 型定義作業中に発覚した追加変更（対応: 型定義書 2章・6.6節・6.7節・6.8節）

型定義書(`仮設計書-型定義.md`)の作成を進める中で、以下の設計変更・新規決定が発生した。本体(`仮設計書.md`)には未反映のため、本設計書化のタイミングで反映すること。

### crypto クレートの新設（対応: 5.3, 5.4 依存関係）

- 当初`notifier`クレート固有だった`webhook_url`暗号化型(`EncryptedWebhookUrl`/`PlainWebhookUrl`)を、**管理者向け通知先設定(下記`system_notify_config`)でも同様の暗号化が必要になったため汎用化**し、新設の`crypto`クレートに切り出した
- 汎用型は`Encrypted<T>`/`Plain<T>`というジェネリクス構造体とし、型パラメータ`T`を「用途タグ」(フィールドを持たないマーカー型、例: `WebhookUrlTag`、`SystemNotifyWebhookUrlTag`)として使うことで、異なる用途の暗号化文字列を型レベルで混同できないようにしている
- **依存関係(5.4節)への追記**:
  - `crypto -> 外部crateのみ`(新規クレート)
  - `notifier -> crypto`(追加)
  - `subscription -> crypto`(追加、後述の`system_notify_config`のため)
- 暗号化方式自体(AES-256-GCM、nonce || ciphertextのbase64、12.1節)は変更なし。型の置き場所と抽象化レベルのみの変更

### system_notify_config テーブルの新設（対応: 13章 DDL、14.7 管理者API）

管理者向けAPI(`GET`/`PUT /api/admin/notify-config`、14.7節)の型定義中に、「定期実行ロガー(10.4節)の通知先設定」に対応するDBテーブルが仮設計書に存在しないことが発覚した。以下の方針で新規追加する。

**検討経緯**
- 「env(環境変数)で設定を持ち、フロントエンドでの設定は将来拡張に送る」案も検討したが、14.7節のAPI設計がすでに`GET`/`PUT`(フロントから読み書き可能)を前提にしていたため、DBに寄せる方が手戻りが少ないと判断し、DBテーブルとして新設することにした
- 当初「`id BOOLEAN PRIMARY KEY DEFAULT TRUE CHECK (id = TRUE)`でシステム全体1行のみを保証する」設計を検討したが、**管理者が複数人いる場合、互いの設定が上書きされ合う**という問題が発覚
- 最終的に「**管理者全体で共有する1つの設定**」として運用する方針で確定。UI上では「共通設定」であることを明示し、誤解を防ぐ。**管理者ごとの個別設定(複数行化)は将来拡張(18章)に持ち越す**

**DDL(新規)**
```sql
-- 定期実行ロガー(MemoryLayer、10.4節)の通知先設定。
-- 管理者全体で共有する1行のみを運用する（複数管理者対応は将来拡張、18章）。
CREATE TABLE system_notify_config (
    id BOOLEAN PRIMARY KEY DEFAULT TRUE CHECK (id = TRUE), -- 常に1行のみを保証するテクニック
    medium notify_medium NOT NULL DEFAULT 'discord',
    webhook_url TEXT,       -- 暗号化して保存（cryptoクレート、12.1節と同じ方式。用途タグはSystemNotifyWebhookUrlTag）
    mention_enabled BOOLEAN NOT NULL DEFAULT FALSE,
    mention_targets TEXT[] NOT NULL DEFAULT '{}',
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

- `medium`/`mention_targets`は`notify_discord_configs`/`notify_slack_configs`(13.4節)と同じ形式を踏襲する
- 型定義は`subscription`クレート側に`SystemNotifyConfig`として配置する(グループに紐づかないシステム全体の設定のため)

### logs関連ドメイン型の配置方針（対応: 13.1, 14.7）

- `logs`テーブル(13.1節)の`level`(`LogLevel`)・`process`(`LogProcess`)・行全体(`LogEntry`)は、`logging`クレートにドメイン型として配置する方針とした
- 管理者ログ一覧API(`GET /api/admin/logs`)のレスポンスには、13.1節の`fields`(JSONB、構造化ログの中身)も**含める**方針とした。フロントエンドにそのまま生のJSONを渡し、表示の作り込みはフロントエンド側に委ねる
- あわせて、クエリパラメータとして14.7節に明記のない`level`/`process`によるフィルタも追加する(13.1節でインデックス`idx_logs_process`が用意されている実態に合わせた)

### 決算情報エクスポートをCSVからExcel(xlsx)へ変更（対応: 14.8）

- 14.11節「フィルタデータのExcel出力」で決定した経緯(文字コード起因の文字化け、`ticker`のような先頭ゼロを含む値が数値として誤解釈される「列の型崩れ」リスク)は、`ticker`列を持つ決算情報エクスポートにも同様に当てはまるため、`GET /api/earnings/export`もフィルタエクスポートと同じ`format`クエリパラメータ方式(MVPでは`xlsx`のみ、`csv`は将来拡張)に変更する

- `/api/earnings/summary`(決算発表日ごとの件数集計)は、タイムゾーン方針(原則UTCのまま返す)の例外として、**日付のみJST基準で切り出す**こととした。レスポンスのフィールド名を`dateJst`とし、JST基準であることを明示する

### notify_queue/notify_history関連ドメイン型の配置と依存関係追加（対応: 型定義書4章・6.8節、14.9）

- `NotifyStatus`/`NotifyQueueEntry`/`NotifyHistoryEntry`は、新規クレートを増やさない方針のもと、**`subscription`クレートに配置**することにした(「送信管理」という関心事が`subscription`の責務に近いため)
- これに伴い**依存関係(5.4節)に`subscription -> earnings`を新規追加**する。`NotifyQueueEntry`/`NotifyHistoryEntry`が`earnings`由来の型(`EarningsEvaluation`/`EarningsSource`)や非正規化列(`ticker`/`company_name`等)を持つため

- `GET /api/notify-history`のレスポンスには、`notify_groups.name`をJOIN済みで含める(`group_name`)方針とした。フロント側でのグループ名解決のための追加リクエストを不要にする(拡張性重視)
