# EarningsWatch 本設計書作成 引き継ぎメモ

## これまでの経緯

「EarningsWatch」(決算速報をDiscord/Slackへ通知する個人・家族・友人向けシステム)は、以下の順で設計を進めてきた。

1. 仮設計書(`仮設計書.md`)の作成、TODO全8件のうち解消可能なものを解消
2. 仮設計書作成後に思いついた追加仕様の検討・仕様策定
3. 型定義方針の決定
4. **クレート単位・エンドポイント単位での厳密な型定義**(`仮設計書-型定義.md`として作成完了)
5. 型定義作業と並行して発覚した追加のDDL変更・仕様変更・依存関係変更を記録(`仮設計書からの追加点と変更点.md`に追記完了)
6. 3ファイル全体を通しての整合性チェック完了(型定義書8章に整合性チェック結果を記録済み)

**現在地点: 上記3ファイルの内容を正として、これらを統合した「本設計書」を新規に作成するフェーズに入る。**

## 本設計書作成にあたって、ユーザーと合意した3つの方針

### 1. 本設計書は章ごとに複数のMarkdownファイルに分割する

1本の巨大なMarkdownにはせず、実装時に参照しやすいようフォルダ構成+複数ファイルに分割する。分割案(たたき台、実際に手を動かす前に細部をユーザーとすり合わせること):

```text
design/
├── 00-overview.md       (仮設計書1〜5章相当: 概要・アーキテクチャ・依存関係。crypto/content新設を反映した依存関係図もここ)
├── 01-db-schema.md      (仮設計書13章相当: DDL全体。system_notify_config/pagesテーブル等の新規DDLも統合)
├── 02-types/
│   ├── identity.md
│   ├── crypto.md
│   ├── earnings.md
│   ├── subscription.md
│   ├── notifier.md
│   ├── content.md
│   └── api.md            (14.5〜14.13節相当のDTO一式)
├── 03-features/          (仮設計書6〜10章相当: 機能ごとの業務ロジック・追加仕様)
├── 04-security.md        (仮設計書11〜12章相当: 認証・暗号化)
└── 05-future-work.md     (仮設計書18章相当: 将来拡張の一覧)
```

### 2. フロントエンドの設計は本設計書のスコープ外であることを明記する

バックエンドが完成した後に別途設計する方針。本設計書のどこか(概要・基本方針にあたる章)に「本書はバックエンドの設計に限定し、フロントエンドの実装・設計はバックエンド完成後に別途行う」旨を明記すること。型定義書側のDTO(`camelCase`変換等)はフロントとの連携を前提にした設計になっているため、その旨も併せて触れておくとよい。

### 3. 本設計書作成は、引き続きClaudeが担当する

このメモを渡された新しいチャット(またはこのチャットの続き)で、Claudeが本設計書の作成作業そのものを行う。ユーザーは方針の意思決定・レビューを行う立場。

## 本設計書作成を始める際の進め方(提案)

1. まず3ファイルを読み込み、全体像を再確認する
2. 上記の分割案(たたき台)をユーザーに提示し、細部(章の境界、ファイル名等)をすり合わせる
3. `仮設計書.md`本体のTODO一覧(19章)で「未解決」のままになっている**TODO#9(お知らせ板・パッチ通知の設計)を「解決済み」に更新**するのを忘れないこと(型定義書8.7節に申し送り事項として記載済み)
4. 分割方針が固まったら、ファイルごとに順次作成していく(仮設計書の型定義作業と同様、1ファイルずつユーザーと合意しながら進めるのが望ましい)

## 特に注意すべき申し送り事項(型定義書8章より抜粋)

- **依存関係(5.4節)への追記が必要**: `crypto -> 外部crateのみ`、`notifier -> crypto`、`subscription -> crypto`、`subscription -> earnings`、`content -> identity`、`api -> content`
- **DDL変更が必要**: `notify_queue`への`is_monitor_marker`列追加・`fingerprint`のNULL許容化、`system_notify_config`テーブル新設、`pages`テーブル新設・`page_type`専用enum型新設
- **`notify_groups`の`ON DELETE CASCADE`見直し**(仮設計書本体でのTODO持ち越し)は、本設計書作成時に確定させる必要がある。確定した場合、型定義書の`NotifyHistoryResponse.group_name`(お知らせ板とは無関係、送信履歴APIのJOIN結果)の扱いに影響する可能性がある
- `SlackConfig`(`notifier`クレート)は仮実装のまま。TODO#1により、Discord実装完了後のMVP内拡張フェーズで再定義する方針
