# EarningsWatch 設計書(ブレスト段階)

## 目的
EarningsWatchは、個人・友人・家族向けの、上場企業等の決算速報をDiscordなどに送信・通知するシステムである。
監視処理は単発実行型であり、定期実行は cron / supercronic など外部スケジューラに委ねる？
監視といっても、まとめ/フィルターのイメージに近く、1日に2回程度の定期送信を想定している。
各設定を個人でもてるようにし、送信内容が分かれるようにする。
また、インターフェースの概念を大切にし、通知対象やスクレイピング対象の変更等が容易にできるようにする練習も兼ねる。

## 仕様技術
- フロントエンド: React
- バックエンド: Rust (Cargo Workspace)
- HTTP API: axum
- スクレイピング: python (Playwirte)
- DB: PostgreSQL
- DBアクセス: sqlx
- コンテナ: Docker / Docker compose
- 認証: JWT + HttpOnly Cookie + Refresh Tokenローテーション
- パスワードハッシュ: argon2

## メモ
スクレイピングはRustで行わず、pythonを使用する。
PlayWirteを使って、より自然に開くことが出来たりするため。

## 定期送信について
目的より、
```
監視処理は単発実行型であり、定期実行は cron / supercronic など外部スケジューラに委ねる。
監視といっても、まとめ/フィルターのイメージに近く、1日に2回程度の定期送信を想定している。
```
そのため、スクレイピング行為は30分前などに時間をかけて安全に行い、送信時間が実行時間によって変わらないようにする設計をする。
したがって、エントリポイントは以下の4つを想定する。

- server: web処理などのサーバの立ち上げ(axum)
- monitor: 定期監視処理。スクレイピングをして送信内容全体を正規化
- notify: 各送信処理。送信内容をフィルタ。それぞれに通知をする。
- cli: migrationや管理者ユーザ作成などのcliコマンドを実行

それかどうせサーバを立ち上げておくなら定期実行もcronでいいかもしれないため、

- server: web処理。定期監視処理。定期送信処理。
- cli: migrationや管理者ユーザ作成などのcliコマンド

という2エントリポイントでもいいかもしれない。エントリポイントをどうするかは要検討


## エントリポイントの検討結果

## スクレイピングについて(一覧→個別ページ問題の検討結果)
サイトは`一覧→個別ページ`のように遷移する必要があるが、決算件数は多いため、
個別ページに毎回遷移するのは処理時間・サイト負荷の両面でコストが高い。
そのため、一覧ページの段階で新規/既知の判別を行い、個別ページへは新規分のみ遷移する設計とする。

### 差分検出の方式: fingerprint
一覧ページで取得できる情報(証券コード・公開日時・決算タイトルなど)のみを使い、
専用の構造体からハッシュ化した値を`fingerprint`として、新規/既知の判別キーとする。

- fingerprintの生成(結合・正規化・ハッシュ化)は **Rust側の1つの関数に集約** する。
  表記揺れによる同一決算の重複判定を防ぐため。
- 個別ページ側ではfingerprintを再計算しない。fingerprintは一覧取得時点で確定する
  「不変な識別子」として扱う。
- DBには決算情報の履歴と並行してfingerprint(ハッシュ化後の文字列)を保存する。

### Rust/Python間の責務分担
Python(Playwright)は生データの取得に徹し、ロジックはRust側に集約する。

- Python: 一覧ページの生データ(判別に使うフィールド群、個別ページへのURL)をJSONで返すのみ
- Rust: 生データからfingerprintを組み立て・ハッシュ化し、DBの既知fingerprintと
  突き合わせて新規/既知を判別する

これにより、正規化ルールの二重管理を避ける。

### 差分検出クエリ(時間ベース/件数ベースの2段階)
新規判別に使う「既知fingerprint」の取得範囲は、決算集中期(3月・11月など)の
件数急増を考慮し、以下の2段階とする。

1. まず時間ベースで取得する(例: 直近7日分)
2. 件数がn件に満たない場合、件数ベースで取得し直す(例: 直近n件)

件数ベースのみだと決算集中期に本来直近の対象がHashSetから漏れるリスクがあるため、
時間ベースを優先し、閑散期の安全マージンとして件数ベースを併用する。
取得は2回のクエリに分けて行う想定。

### ページ送り制御
一覧が1ページに収まらず2ページ目以降に続く可能性があるため、ページ送りの
継続判定を以下のように定める。

- 該当ページの全件が新規と判定されたら、次ページを取得する
- 既知のfingerprintが1件でも検出されたら、その時点でページ送りを打ち切る

制御ループはRust(`monitor`)側が持ち、Pythonは`list --page N`のように
ページ番号を受け取って該当ページの生データを返すだけの役割とする
(スクレイピング特有の状態やロジックをPython側に持たせない)。

### 処理フロー(まとめ)
```
1. Rust(monitor)がPythonに `list --page 1` を実行させる
2. Rustが返却された生データからfingerprintを組み立てる
3. DBから既知fingerprint(時間ベース→不足時は件数ベース)をHashSetとして取得
4. 一覧のfingerprintとHashSetを突き合わせ
   - 全件新規 → 次ページを取得(1に戻る、pageをインクリメント)
   - 既知が1件でもあれば → ページ送りを打ち切り、新規分を確定
5. 新規分のURLのみ、個別ページへ遷移して詳細情報を取得
6. 個別ページの内容と合わせて `MonitoredEarningsReport` として正規化する
```

## ロギングとその送信
2つのロガーを用意する。

- 全体のロガー。infoも含む。フロントエンドで確認できるようにする。SQL？
- 定期実行時のロガー。warnやerrorが中心。メモリに保存し、確認用にdiscordなどに送信を行う。その後メモリは解放される。

定期実行のロガー設定は管理者ユーザのみが設定できるようにするイメージである。


## ロギングとその送信(検討結果)
`tracing`クレートを用い、ログの発生箇所(呼び出し側)と、その処理方法(保存・通知)を
Layerとして分離する設計とする。呼び出し側は`info!`/`warn!`/`error!`などのマクロで
イベントを発行するだけでよく、保存先や通知先を意識しない。

### Layer構成
- `SqlLayer`: ログをSQLに保存する(全体ロガーに相当)
- `MemoryLayer`: warn/error以上をメモリに貯め、管理者が設定した通知先(Discordなど)へ
  送信するトリガーを持つ(定期実行時のロガーに相当)

登録するLayerはエントリポイントによって異なる。

- `server`起動時: `SqlLayer`のみ登録
- `cli monitor` / `cli notify`起動時: `SqlLayer` + `MemoryLayer`を登録

これにより、`logger`側の実装(SQL書き込み、通知送信)は各Layerの内部に閉じ込められ、
呼び出し側やLayerを束ねる初期化処理はSQL/通知の実装詳細を知らずに済む。

### SQL保存(`SqlLayer`)の書き込み方式
`on_event`は同期関数であり、中で直接`.await`はできない。また、ログはイベントの
発生頻度が高くなりうるため、都度同期的にDB書き込みへ行く設計はレイテンシ・負荷の
両面で不利。そのため、以下の非同期バッチ書き込み方式とする(YaakoDNSのクエリログと
同様の構成)。

- `on_event`はログエントリをチャネルに送るのみ(非ブロッキング)
- 別タスクがチャネルを受信し、メモリバッファに蓄積した上でPostgreSQLへバルクINSERTする

flushする条件は以下の通り。

- n件溜まったら
- (`server`のみ)フロントエンドからログ表示のリクエストが来たら
- (`cli monitor` / `cli notify`のみ)単発実行が終わったら(最終flush)

`server`と`cli`は別プロセスであるため、バッファ・flushタスクもプロセスごとに
独立して存在する(意図的に2種類設計するというより、プロセスが分かれている結果
自然とそうなる)。

### ログエントリに含める情報と、その取得方法
`tracing`のマクロを呼ぶだけで自動的に取れる情報と、Layer実装側で自分で
取得・付与する必要がある情報がある。

| 情報 | 取得方法 |
|---|---|
| ログレベル | `event.metadata().level()`から自動取得 |
| 発生箇所(ファイル/行/モジュール) | `event.metadata()`から自動取得 |
| 発生時刻 | 自動では付与されないため、Layer側で`Utc::now()`等を取得しセットする |
| メッセージ・任意のフィールド内容 | `Visit`トレイトを実装し、`event.record()`で走査して組み立てる |
| どのプロセス由来か(server/monitor/notify) | `tracing`にはない概念のため、Layer初期化時に
  プロセス種別を固定値として持たせ、ログエントリに付与する |

「どのプロセス由来か」は、DBのログテーブル設計時にカラムとして明記する。

### 通知(`MemoryLayer`)の送信先について
定期実行時(`cli monitor` / `cli notify`)にwarn/error以上が発生した場合、
実行終了時にメモリバッファの内容を、管理者がログ通知設定で指定した通知先
(Discordなど)へ送信する。通知先は前述の「定期実行のロガー設定は管理者ユーザ
のみが設定できる」の枠組みで管理する。

また、`server`側で見逃せない重大なエラーが発生した場合も、同じ通知先へ送信する
経路を用意しておく。ただし`server`は常駐プロセスであり「実行終了時にflush」という
`cli`側のトリガーは適用できないため、`MemoryLayer`は以下のように責務を分けて設計する。

- **バッファリング責務**: warn/error以上をメモリに貯める(`server`/`cli`共通)
- **flushトリガー責務**: いつ通知先へ送るかはプロセスの性質によって異なるため、
  外部から注入できるようにしておく
  - `cli`(monitor/notify): プロセス終了時にflush
  - `server`: error以上の発生時に即時、または短い時間窓でflush(詳細は実装時に検討)

### 不明点(メモ)
- `server`側のflushトリガー(即時か、短時間窓か)は未決定。実装時に検討する。

### ロギング
infoなどを含めたログを蓄積する用。
フロントエンドなどからも確認したいためSQL化する。
tracing-subscriberのlayerとして実装するため、tracingの内容の一部を保存していくイメージ。
どのプロセス(server/monitor/notify)由来かを明記するカラムを持たせる。

```sql
CREATE TYPE log_process AS ENUM (
    'server',
    'monitor',
    'notify'
);

CREATE TABLE logs (
  -- 自動採番ID
  id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,

  -- ログの発生時刻
  timestamp   TIMESTAMPTZ NOT NULL,

  -- ログレベル
  level       VARCHAR(5) NOT NULL
    -- ログ項目のチェック
    CHECK (level IN (
      'TRACE',
      'DEBUG',
      'INFO',
      'WARN',
      'ERROR'
    )),

  -- どのプロセス由来か
  process     log_process NOT NULL,

  -- ログ発生場所
  target      TEXT NOT NULL,

  -- メッセージ
  message     TEXT,

  -- 構造化ログ用
  fields      JSONB NOT NULL DEFAULT '{}'::jsonb
);

CREATE INDEX idx_logs_timestamp ON logs (timestamp DESC);
CREATE INDEX idx_logs_process ON logs (process);
```

サブタイプは、sqlxの型安全を利かせるため、媒体ごとに専用テーブルを作る方針とする。
また、処理上は証券コードと銘柄名の片方が埋まっていればいいが、見た目上どっちも必須とする。
フロントエンドでグループの送信媒体などの設定を切り替えた際、設定を残すため、実際にははなからすべてのテーブルを持っているような設計にする(グループ作成時に必ず1行作成し、以後残し続ける)。
また、グループ一括設定などの項目は持たせず、フロントエンド側で、一括設定をするような機能を持たせるだけとする。
また、SlackなどはtoDoとなっているため後で調べる。

-- ユーザ個人設定

### 送信内容
monitor処理で取得したnotify処理で送信するべき内容を入れるテーブル。
ソースなどは重複するが、非正規化した方が楽なので非正規化する。
これらはmonitor処理をするたびに削除・変更される。
`earnings`テーブルとの対応が追跡できるよう、`fingerprint`を参照として持たせる。
```sql
CREATE TYPE notify_status AS ENUM (
    'ready',  -- 送信準備完了
    'sent',   -- 送信済み
    'failed'  -- 送信失敗
);

CREATE TABLE notify_queue (
  -- 自動採番ID
  id BIGSERIAL PRIMARY KEY,

  -- earningsテーブルとの対応関係を追跡するための参照
  fingerprint TEXT NOT NULL REFERENCES earnings(fingerprint),

  -- 取得元サイト
  source earnings_source NOT NULL,

  -- 取得時間(スクレイピング開始時間)
  fetched_at TIMESTAMPTZ NOT NULL,

  -- 証券コード
  ticker TEXT NOT NULL,
  -- 銘柄名
  company_name TEXT NOT NULL,
  -- 公開時刻
  published_at TIMESTAMPTZ NOT NULL,
  -- 決算タイトル
  title TEXT NOT NULL,
  -- 詳細情報へのurl
  url TEXT NOT NULL,
  -- 内容の要約など
  summary TEXT NOT NULL,
  -- 決算評価
  evaluation earnings_evaluation NOT NULL,
  -- 送信したか否か等
  status notify_status NOT NULL DEFAULT 'ready'
);

CREATE INDEX idx_notify_queue_status ON notify_queue (status);
```

## 設計の一貫性に関わる点

**1. `notify_groups`とフィルタ/グループ削除時のnotify_queueの扱い**
グループやフィルタを削除した場合、`notify_queue`に既に溜まっている「まだ送信していない」行はどうなるか。`notify_queue`は`fingerprint`経由で`earnings`は参照していますが、`notify_groups`や`notify_filters`への参照は持っていません。つまり「どのユーザ・どのグループ向けか」を`notify_queue`自体は知らない設計になっています。これは意図的でしょうか?(notify処理の実行時に、毎回全ユーザの全フィルタと`notify_queue`を突き合わせる、という設計を想定している?)

**2. 「グループ一括設定」の対象範囲**
以前「全体の一括設定」と「グループ毎の一括設定」という2種類のCSV一括設定がありましたが、今回追加した「送信先の一括設定(フロントエンドの機能)」との関係が整理されていません。CSV一括設定は「フィルタ内容」、フロントエンド一括設定は「送信先」という別物ですが、両方が「一括設定」という言葉を使っているので、混乱しないよう用語を分けておくと良さそうです。

**3. `notify_filters`の重複行の扱い**
同じグループ内で、同じ`ticker`+`company_name`の組み合わせを複数回登録できてしまう(UNIQUE制約がない)。CSV一括設定で重複行が入ってきた場合の挙動(スキップするか、エラーにするか)を決めていないため、ここも考慮点になりそうです。

## 認証・セキュリティ関連

**4. JWT/Cookieの具体的なフロー**
「JWT + HttpOnly Cookie + Refresh Tokenローテーション」という技術スタックは決まっていますが、具体的な有効期限、ローテーションのタイミング(毎回か、期限が近づいたらか)は今の資料にありません。YaakoDriveで既に実装済みとのことなので流用で済むかもしれませんが、「今回のプロジェクトでも同じ設計をそのまま使う」と明記しておくと良さそうです。

**5. 管理者ユーザとそれ以外のユーザの権限差**
`users.role`カラムはあるものの、「管理者のみができること」(定期実行のロガー設定、CLIでのcreate-adminなど)以外に、権限による機能制限がどこまであるか整理されていません。個人・友人・家族向けとのことなので、シンプルな2階層(admin/user)で足りるのか、確認しておくと良さそうです。

## 運用・デプロイ関連

**6. マイグレーション実行のタイミング**
`cli migration`は「初期化時に実行する」とありますが、今後スキーマ変更が発生した際の運用フロー(自動実行か、手動でexecして都度流すか)を軽くメモしておくと、デプロイ時に迷わなくて済みます。

**7. Python(Playwright)環境のコンテナ配置**
コンテナ設計は「Frontend/Backend/DB」の3つとのことですが、PythonのPlaywright実行環境がどこに入るか明記されていません。Backendコンテナ内にPythonランタイムも同居させるのか、Playwrightは別途ヘッドレスブラウザの依存(Chromiumなど)が重いので、その点も考慮が要りそうです。

**8. スクレイピング対象サイトの利用規約・robots.txt確認**
技術的な話ではないですが、個人利用とはいえ「スクレイピング行為」を継続的に行うシステムなので、対象サイトの利用規約やアクセス頻度の妥当性(1日2回程度なら問題になりにくいとは思います)を一度確認しておくと安心です。

## 機能面で決まっていない点
