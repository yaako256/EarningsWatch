## 更新版API一覧

### その他
| パス | 内容 |
| --- | --- |
| `GET /api/health` | ヘルスチェック |

### 認証関連
| パス | 内容 |
| --- | --- |
| `POST /api/auth/login` | ログイン |
| `POST /api/auth/refresh` | トークンのリフレッシュ |
| `POST /api/auth/logout` | ログアウト |
| `GET /api/auth/me` | 再訪問時の自動ログイン用 |

### 管理者ユーザ用
| パス | 内容 |
| --- | --- |
| `GET /api/admin/logs?page=&per_page=` | ログ一覧(オフセット型ページング) |
| `GET /api/admin/users?page=&per_page=` | ユーザ一覧(オフセット型ページング) |
| `POST /api/admin/users/{id}/disable` | ユーザ無効化 |
| `GET /api/admin/users/{id}/summary` | 特定ユーザのグループ数/フィルタ数/媒体種別の集計 |
| `GET /api/admin/notify-config` | 定期実行ロガーの通知先設定取得 |
| `PUT /api/admin/notify-config` | 定期実行ロガーの通知先設定更新 |

### 決算情報関連
| パス | 内容 |
| --- | --- |
| `GET /api/earnings?page=&per_page=&ticker=` | 決算情報一覧(ページング、tickerフィルタ可) |
| `GET /api/earnings/export` | 決算情報一覧のCSVエクスポート |

### グループ関連
| パス | 内容 |
| --- | --- |
| `GET /api/groups` | 自分のグループ一覧 |
| `POST /api/groups` | グループ作成 |
| `PATCH /api/groups/{id}` | グループ名・媒体変更 |
| `DELETE /api/groups/{id}` | グループ削除 |
| `GET /api/groups/{id}/config` | 媒体別設定取得 |
| `PUT /api/groups/{id}/config` | 媒体別設定更新 |
| `PUT /api/groups/bulk-destination` | 送信先一括設定 |

### フィルタ関連(グループ配下ネスト)
| パス | 内容 |
| --- | --- |
| `GET /api/groups/{id}/filters?page=&per_page=` | グループ内フィルタ一覧 |
| `POST /api/groups/{id}/filters` | フィルタ追加 |
| `PATCH /api/groups/{id}/filters/{filter_id}` | フィルタ有効化/無効化・備考変更 |
| `DELETE /api/groups/{id}/filters/{filter_id}` | フィルタ削除 |
| `POST /api/filters/import` | フィルタ一括インポート(全体設定) |
| `POST /api/groups/{id}/filters/import` | フィルタ一括インポート(グループ単位) |
| `GET /api/filters/export` | フィルタ内容のCSVエクスポート |

### ユーザ設定
| パス | 内容 |
| --- | --- |
| `GET /api/users/me/settings` | 自分の設定取得 |
| `PATCH /api/users/me/settings` | 自分の設定更新 |

### ダッシュボード
| パス | 内容 |
| --- | --- |
| `GET /api/dashboard` | 監視銘柄数・直近送信状況などの集計 |

---

次に決めるべきこととして、いくつか出てきそうな論点があります。

- `PATCH /api/groups/{id}/filters/{filter_id}`のような更新系で、部分更新(有効化のみ変更 等)を`PATCH`1本にまとめるか、`POST /api/groups/{id}/filters/{filter_id}/enable`のように操作ごとに分けるか
- フィルタ一括インポート(`POST /api/filters/import`)のレスポンス形式(成功件数・エラー件数・重複件数をどう返すか)

このあたりはAPI設計というよりリクエスト/レスポンスの詳細設計になってくるので、一通りのパス一覧が固まった段階でまとめて詰めるのがいいと思います。他に追加したいリソース(通知履歴の詳細など)はありますか、それとも一旦このAPI一覧を`ブレスト段階設計書ver3`としてまとめますか?
